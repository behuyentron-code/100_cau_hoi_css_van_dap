Phần 1 — Flexbox
1 .top-menu dùng display: flex; justify-content: space-between — giải thích justify-content và align-items khác nhau như thế nào? Flexbox▸

justify-content phân bố items theo trục chính (main axis — ngang khi flex-direction: row). align-items căn chỉnh theo trục phụ (cross axis — dọc). Menu dùng space-between để logo ở trái, auth-group ở phải; dùng align-items: center để chúng đều nằm giữa theo chiều dọc.

2 .cart-layout { display: flex; gap: 28px; align-items: flex-start } — tại sao cần align-items: flex-start? Nếu không có thì sao? Flexbox▸

Mặc định align-items: stretch — cả 2 cột (items và summary) sẽ bị kéo dài bằng nhau. flex-start cho phép mỗi cột cao đúng bằng nội dung của nó, để position: sticky trên order summary mới hoạt động đúng.

Liên quan: nếu align-items là stretch, sticky sẽ không có tác dụng vì phần tử đã bị kéo dài hết cột rồi.
3 .summary-col { flex: 0 0 320px } — ba giá trị của flex shorthand là gì? Tại sao dùng 0 0? Flexbox▸

flex: grow shrink basis. flex: 0 0 320px nghĩa là: không co giãn (grow=0), không thu nhỏ (shrink=0), luôn rộng đúng 320px. Sidebar tóm tắt đơn hàng cần kích thước cố định, không bị ảnh hưởng khi danh sách sản phẩm dài hay ngắn.

4 .cart-items-col { flex: 1; min-width: 0 } — min-width: 0 có tác dụng gì? Không có thì bị lỗi gì? Flexbox▸

Flex item mặc định có min-width: auto — nghĩa là không bao giờ thu nhỏ hơn nội dung bên trong. Nếu tên sản phẩm dài, cột sẽ tràn ra ngoài container. min-width: 0 bỏ giới hạn đó, cho phép cột co lại khi cần.

Đây là bug rất hay gặp khi dùng flex: 1 nhưng cột vẫn bị tràn.
5 .action-row { display: flex; gap: 12px } với 2 nút đều có flex: 1 — kết quả layout là gì? Flexbox▸

Cả 2 nút "Thêm vào giỏ" và "Mua ngay" sẽ chia đều không gian, mỗi nút chiếm 50% chiều rộng (trừ gap). Dù text dài hay ngắn, 2 nút luôn bằng nhau — giao diện đối xứng.

6 .features-row { flex-wrap: wrap } — flex-wrap làm gì? Khi nào nó xuống dòng? Flexbox▸

Mặc định flex items không xuống dòng, bị ép vào một hàng dù tràn. flex-wrap: wrap cho phép items xuống dòng khi tổng chiều rộng vượt container. Ở trang chi tiết sản phẩm, các feature item (bảo hành, miễn phí vận chuyển...) sẽ tự xuống dòng trên màn hình nhỏ.

7 .qty-control { gap: 0; overflow: hidden; border-radius: 10px } — tại sao cần gap: 0 và overflow: hidden kết hợp với nhau? Flexbox▸

Nhóm nút +/- và input được thiết kế liền nhau, không có khoảng cách giữa. gap: 0 loại bỏ khoảng hở giữa các flex item. overflow: hidden kết hợp border-radius để bo góc cả nhóm — các góc của button bên trong tự bị cắt theo border-radius của container cha.

8 .detail-wrapper { display: flex; gap: 48px; align-items: flex-start } — nếu đổi thành flex-direction: column thì layout bị ảnh hưởng gì? Flexbox▸

Ảnh và thông tin sản phẩm đang đứng cạnh nhau (hàng ngang). Đổi thành column sẽ xếp ảnh trên, thông tin dưới. Đây thực ra là layout mong muốn trên mobile — nhưng code hiện tại chưa có media query để làm điều này.

9 .voucher-card { display: flex; overflow: hidden } — overflow: hidden trên flex container có ảnh hưởng gì đến position: sticky của các flex item bên trong không? Flexbox▸

Có — đây là bẫy rất phổ biến. overflow: hidden/auto/scroll trên cha phá vỡ position: sticky của con vì nó tạo ra một scroll container mới. Voucher card không có sticky nên không bị ảnh hưởng ở đây, nhưng đây là điều cần nhớ.

Tương tự với sidebar dùng sticky — cha của nó KHÔNG được có overflow hidden.
10 .coupon-row { display: flex; gap: 8px } với input có flex: 1 và button có white-space: nowrap — giải thích cách layout này hoạt động. Flexbox▸

Input với flex: 1 chiếm toàn bộ không gian còn lại. Button có white-space: nowrap ngăn text "Áp dụng" bị xuống dòng dù container hẹp — button giữ kích thước tự nhiên, input co giãn theo phần còn lại. Pattern kinh điển cho input + button inline.

Phần 2 — CSS Grid
11 .cart-header và .cart-item cùng dùng grid-template-columns: 2fr 1fr 1.2fr 0.7fr 36px — tại sao phải giống nhau hoàn toàn? Grid▸

Để các cột header và data item thẳng hàng với nhau, như một bảng. Nếu 2 cái khác nhau, dữ liệu sẽ lệch khỏi tiêu đề. Đây là pattern phổ biến thay thế <table> bằng CSS Grid để có layout linh hoạt hơn.

12 .stats-grid { grid-template-columns: repeat(4, 1fr) } trong admin.css — nếu chỉ có 3 stat card thì ô thứ 4 trông như thế nào? Grid▸

Ô thứ 4 sẽ trống (empty grid cell). Grid vẫn tạo 4 cột, 3 item điền vào 3 ô đầu, ô cuối bỏ trống. Nếu muốn tự động, dùng repeat(auto-fit, minmax(200px, 1fr)) — grid tự điều chỉnh số cột theo số item.

13 .promo-grid { grid-template-columns: repeat(3, 1fr); gap: 22px } — gap ở đây áp dụng cho cả hàng và cột hay chỉ một chiều? Grid▸

gap: 22px là shorthand áp dụng cho cả row-gap và column-gap. Nếu muốn khác nhau: gap: 16px 22px (row 16, column 22). Còn gap: 22px 0 chỉ có khoảng cách giữa hàng, không có giữa cột.

14 .form-row { grid-template-columns: 1fr 1fr } trong admin.css — khi màn hình nhỏ code đổi thành 1fr. Giải thích logic responsive này. Grid▸

Trên desktop, 2 field form nằm cạnh nhau (1fr 1fr). Khi max-width: 768px, đổi thành 1fr — mỗi field chiếm toàn bộ chiều rộng, xếp dọc. Đây là responsive form pattern chuẩn.

15 .empty-state { grid-column: 1 / -1 } — giá trị -1 có nghĩa là gì trong grid? Grid▸

-1 nghĩa là cột cuối cùng của grid. grid-column: 1 / -1 = span từ cột 1 đến cột cuối = chiếm toàn bộ chiều rộng grid. Dùng cho empty state vì nó cần chiếm hết 3 cột của promo-grid, không phụ thuộc vào số cột cụ thể.

16 .voucher-grid { grid-template-columns: repeat(3, 1fr) } — nếu có 4 voucher, voucher thứ 4 sẽ ở đâu? Cần thêm gì để chúng luôn đều nhau? Grid▸

Voucher thứ 4 tự động xuống hàng 2, chiếm cột đầu tiên (rộng 1fr). Các ô còn lại trống. Để chúng luôn đều: dùng repeat(auto-fill, minmax(240px, 1fr)) — tự điều chỉnh số cột theo chiều rộng màn hình.

17 .footer-modern { grid-template-columns: 2fr 1fr 1fr 1fr } — tổng cộng bao nhiêu fr? Cột đầu chiếm bao nhiêu % không gian? Grid▸

Tổng 5fr. Cột đầu chiếm 2/5 = 40%. Ba cột còn lại mỗi cột 1/5 = 20%. Đơn vị fr chia phần không gian còn lại SAU khi tính gap.

Phần 3 — Position & Z-index
18 .top-menu { position: sticky; top: 0 } — liệt kê 3 điều kiện để sticky KHÔNG hoạt động. Position▸

1) Cha có overflow: hidden/auto/scroll. 2) Phần tử cha không đủ chiều cao để scroll (cha thấp hơn phần tử sticky). 3) Quên khai báo top/bottom/left/right — thiếu giá trị này thì sticky hoạt động như relative. Trong code, nav có overflow: visible !important để tránh lỗi 1.

19 .order-summary { position: sticky; top: 20px } — tại sao không dùng top: 0? Position▸

Menu sticky đã chiếm khoảng 85px ở trên cùng. top: 0 sẽ đẩy order summary lên sát mép trên viewport, bị menu đè. top: 20px chỉ tạo khoảng đệm nhỏ, nhưng thực ra nên là ít nhất 90-100px để tránh chồng lên menu.

20 .badge { position: absolute; top: -5px; right: -5px } — điều kiện gì PHẢI có trên phần tử cha để badge hiển thị đúng vị trí? Position▸

Cha (.icon-btn) phải có position: relative/absolute/fixed/sticky — bất kỳ giá trị nào khác static. Không có → badge định vị theo tổ tiên gần nhất có position, có thể là body → badge xuất hiện sai chỗ hoàn toàn.

21 .modal { position: fixed; z-index: 2000 }, menu z-index: 1000, dropdown z-index: 1000 — tại sao modal cần 2000? Position▸

Modal phải overlay lên MỌI thứ — menu, dropdown, chat sidebar. Nếu modal cũng là 1000, thứ tự hiển thị phụ thuộc vào vị trí trong DOM (ai sau thì trên). Đặt 2000 đảm bảo modal luôn trên tất cả không cần quan tâm DOM order.

Bẫy: z-index chỉ tác dụng khi position không phải static. Modal dùng fixed nên OK.
22 .dropdown-content { position: absolute; top: 100% } — top: 100% có nghĩa gì? Tại sao không phải top: 40px? Position▸

top: 100% = cách từ top của cha 100% chiều cao cha = dropdown bắt đầu ngay bên dưới cha. Nếu dùng 40px hardcode, khi cha thay đổi chiều cao thì dropdown bị chồng hoặc có khoảng trống. 100% luôn chính xác bất kể chiều cao cha.

23 .featured-ribbon { position: absolute; top: 18px; left: 18px } — cha của nó là .featured-deal { position: relative }. Nếu xóa position: relative trên cha thì ribbon đi đâu? Position▸

Ribbon sẽ định vị theo tổ tiên positioned gần nhất — có thể là .promo-wrap hoặc body. Ribbon sẽ xuất hiện ở góc trên trái của toàn trang thay vì góc card, hoàn toàn sai vị trí.

24 .discount-badge { position: absolute; top: 12px; right: 12px } trong khuyen_mai.css — badge này hình tròn, làm sao CSS tạo hình tròn từ một div? Position▸

width: 52px; height: 52px; border-radius: 50%. border-radius: 50% trên phần tử vuông (width = height) tạo hình tròn hoàn hảo. Nếu width ≠ height thì ra hình bầu dục (ellipse).

25 .admin-sidebar { position: fixed; height: 100vh } và .admin-content { margin-left: 260px } — tại sao content cần margin-left bằng đúng width của sidebar? Position▸

Sidebar fixed bị tách khỏi document flow — nó không chiếm không gian. Content sẽ bị sidebar đè lên 260px bên trái. margin-left: 260px đẩy content sang phải đúng bằng chiều rộng sidebar để tránh bị che.

Phần 4 — Animation & Transition
26 @keyframes slideDown { from { transform: translateY(-30px); opacity: 0 } to { transform: translateY(0); opacity: 1 } } — tại sao dùng transform thay vì top/margin-top? Animation▸

transform và opacity được xử lý bởi GPU (composite layer), không trigger layout reflow hay repaint. top/margin-top thay đổi sẽ trigger reflow toàn trang → giật lag. Rule ngón tay cái: animate chỉ transform và opacity để smooth nhất.

27 .promo-card:hover { transform: translateY(-6px); transition: .3s } — transition: .3s thiếu gì so với khai báo đầy đủ? Điều này ảnh hưởng gì? Animation▸

Khai báo đầy đủ: transition: property duration timing-function delay. Thiếu property (áp dụng cho all) và timing-function (mặc định ease). Viết .3s shorthand như vậy hợp lệ nhưng không rõ ràng. Nên viết: transition: transform .3s ease, box-shadow .3s ease để chỉ animate những property cần thiết, tránh lãng phí.

28 .btn-checkout:hover { transform: translateY(-2px); box-shadow: 0 8px 24px rgba(78,92,52,.3) } — animation này nên đặt transition ở đâu: trên .btn-checkout hay .btn-checkout:hover? Animation▸

Phải đặt trên .btn-checkout (không phải :hover). Nếu đặt trên :hover, animation chỉ chạy khi hover VÀO (smooth), nhưng khi rời chuột ra sẽ snap ngay về trạng thái ban đầu không có animation. Đặt trên element gốc đảm bảo transition cả 2 chiều vào và ra.

Đây là lỗi cực kỳ phổ biến của sinh viên.
29 .contact-box:hover { transform: translateY(-5px); box-shadow: 0 5px 15px rgba(0,0,0,0.1) } — tại sao hover card dùng translateY(-5px) chứ không phải margin-top: -5px? Animation▸

margin-top: -5px thay đổi layout flow → các phần tử xung quanh bị dịch chuyển (layout shift). translateY(-5px) chỉ di chuyển visual của phần tử, không ảnh hưởng layout flow. Các card bên cạnh không bị xô lệch.

30 .hero-cta:hover { transform: translateY(-2px) } và .btn-deal:hover { transform: translateY(-2px) } — cùng style, sao không tạo một class chung? Animation▸

Đây là vấn đề DRY (Don't Repeat Yourself). Nên tạo utility class .hover-lift:hover { transform: translateY(-2px) } dùng chung. Hoặc dùng CSS custom property. Code hiện tại bị lặp — nếu muốn đổi thành -3px phải sửa nhiều chỗ.

31 .promo-hero::before, .promo-hero::after dùng animation hay transform không? Tác dụng thực sự của chúng là gì? Animation▸

Không dùng animation/transform — chỉ là hình trang trí tĩnh (2 hình tròn mờ). ::before là hình tròn lớn góc phải trên, ::after là hình tròn nhỏ hơn bên trái dưới. Tạo cảm giác chiều sâu cho banner mà không cần ảnh thêm. Cần content: '' để pseudo-element xuất hiện.

32 transition: border-color 0.2s trên input — tại sao chỉ animate border-color chứ không phải all? Animation▸

transition: all animate MỌI property thay đổi — kể cả những thứ không muốn như width, height, padding khi browser render. Gây lãng phí performance. Chỉ specify đúng property cần animate là best practice.

Phần 5 — Pseudo-class & Pseudo-element
33 .search-container:focus-within — pseudo-class này khác :focus như thế nào? Tại sao không dùng :focus trực tiếp? Pseudo▸

:focus chỉ áp dụng khi chính phần tử đó được focus. Div không focusable nên :focus trên .search-container (div) không bao giờ active. :focus-within active khi bất kỳ descendant nào được focus — khi user click vào input bên trong.

34 .spec-table tr:nth-child(even) td { background: #f0f3ea } — nth-child(even) tô màu hàng nào? Viết tương đương bằng nth-child(2n)? Pseudo▸

Tô màu hàng 2, 4, 6, 8... (chẵn). nth-child(even) = nth-child(2n) hoàn toàn tương đương. Ngược lại nth-child(odd) = nth-child(2n+1) tô hàng lẻ. Pattern này gọi là "zebra striping" — giúp dễ đọc bảng dữ liệu dài.

35 .left-menu ul li::before { content: "›" } — phân biệt ::before (2 dấu ::) và :before (1 dấu :) — cái nào đúng chuẩn CSS3? Pseudo▸

CSS3 chuẩn dùng :: (double colon) cho pseudo-element, : (single) cho pseudo-class. Lý do: phân biệt rõ ràng giữa ::before/::after (tạo element ảo) và :hover/:focus (trạng thái). Browser vẫn chấp nhận single colon vì lý do backward compat nhưng double là chuẩn.

36 .section-title::after { content: ''; flex: 1; height: 2px; background: linear-gradient(...) } — tại sao pseudo-element này hoạt động trong flex context? Pseudo▸

.section-title là flex container (display: flex). Pseudo-element ::after trở thành flex item như các con khác. flex: 1 khiến nó chiếm toàn bộ không gian còn lại sau tiêu đề và icon — tạo đường kẻ ngang tự động giãn ra.

37 .filter-tab:hover:not(.active) — giải thích selector kép này. Tại sao cần :not(.active)? Pseudo▸

Áp dụng hover style CHỈ KHI tab chưa active. Nếu không có :not(.active), hover style sẽ override active style khi user hover lên tab đang active → active tab bị mất style. :not() là pseudo-class loại trừ.

38 .voucher-left::after { content: ''; position: absolute; right: -8px; border-radius: 50% } — pseudo-element này tạo hình gì? Mục đích thẩm mỹ là gì? Pseudo▸

Tạo hình tròn nhỏ ở mép phải của phần màu tối voucher — giả lập hình "vé xé" (notch). Kết hợp với border dashed của card, tạo cảm giác voucher có thể xé rời. Đây là ví dụ dùng pseudo-element cho trang trí thẩm mỹ thuần túy.

39 .admin-sidebar li a:hover { background: #4e5c34; color: white } — selector này có độ ưu tiên (specificity) là bao nhiêu? Pseudo▸

Specificity: .admin-sidebar(class=0,1,0) + li(tag=0,0,1) + a(tag=0,0,1) + :hover(pseudo-class=0,1,0) = (0, 2, 2) = 022. Cao hơn style inline thì không thắng, nhưng thắng được các selector ít class hơn.

40 .tab-btn.active — đây là gì? Khác .tab-btn .active (có dấu cách) như thế nào? Pseudo▸

.tab-btn.active (không dấu cách) = phần tử có CẢ HAI class "tab-btn" VÀ "active". .tab-btn .active (có dấu cách) = phần tử class "active" là CON của phần tử class "tab-btn". Khác nhau hoàn toàn — lỗi thêm/bỏ dấu cách là bug rất hay gặp.

Phần 6 — Responsive & Media Query
41 <meta name="viewport" content="width=device-width, initial-scale=1.0"> — thẻ này có trong tất cả các trang. Nếu không có thì mobile hiển thị như thế nào? Responsive▸

Không có viewport meta, mobile browser giả lập màn hình desktop (thường 980px) rồi zoom out để fit. Toàn bộ trang bị thu nhỏ, chữ rất nhỏ, user phải pinch-zoom. Media query max-width: 768px cũng sẽ không trigger vì browser nghĩ màn hình rộng 980px.

42 File admin.css có 2 media query: max-width: 768px và max-width: 640px. Đây là mobile-first hay desktop-first? Cách viết nào được khuyến nghị hơn? Responsive▸

Desktop-first — style mặc định cho desktop, dùng max-width để override khi màn hình nhỏ hơn. Mobile-first dùng min-width — style mặc định cho mobile, thêm feature khi màn hình lớn hơn. Mobile-first được khuyến nghị vì trang web ngày nay phần lớn truy cập từ mobile.

43 lienhe.css link bằng đường dẫn css\lienhe.css (dùng backslash \). Đây có phải lỗi không? Trên OS nào sẽ bị lỗi? Responsive▸

Đây là lỗi! Backslash (\) là path separator của Windows. Trên Linux/Mac server (hầu hết production server), chỉ forward slash (/) mới hợp lệ trong URL path. Code chạy được trên Windows localhost nhưng deploy lên Linux server sẽ 404. Luôn dùng css/lienhe.css.

Đây là bug thực tế rất hay gặp khi dev trên Windows deploy lên Linux.
44 .hero-banner có @media (max-width: 768px) { height: 200px }. Toàn bộ web chỉ có responsive ở điểm này — liệt kê ít nhất 3 component khác cần thêm responsive. Responsive▸

1) .product-grid: 3 cột → 1 cột trên mobile. 2) .cart-layout: flex row → flex column trên mobile. 3) .detail-wrapper: ảnh và info cạnh nhau → xếp dọc. 4) footer-modern: 4 cột grid → 1-2 cột. 5) Menu ngang → hamburger menu.

45 .filter-bar input { flex: 1 1 200px } trong admin.css — giải thích 1 1 200px. Tại sao basis là 200px? Responsive▸

flex: 1 1 200px = grow 1 (có thể lớn hơn), shrink 1 (có thể nhỏ hơn), basis 200px (kích thước khởi đầu). 200px là kích thước tối thiểu hợp lý cho input — đủ rộng để user gõ text. Nếu container đủ rộng, input sẽ grow vượt 200px.

46 .filter-bar button { flex: 0 0 auto } — tại sao button dùng 0 0 auto thay vì 1 như input? Responsive▸

Button "Tìm kiếm" không cần giãn — nó có kích thước cố định vừa với label. flex: 0 0 auto nghĩa là không grow, không shrink, giữ kích thước tự nhiên. Nếu dùng flex: 1, button sẽ rộng bằng input — trông rất xấu.

Phần 7 — Typography & Text
47 Toàn bộ project dùng font 'Montserrat' từ Google Fonts. Tại sao phải có font-family: 'Montserrat', sans-serif — sans-serif ở sau có tác dụng gì? Typography▸

sans-serif là fallback font — nếu Montserrat không load được (mạng chậm, offline, blocked), browser dùng font sans-serif hệ thống (Arial, Helvetica...). Luôn khai báo fallback để tránh browser dùng Times New Roman (serif mặc định) gây ra giao diện xấu.

48 .hero-title { font-size: 50px; font-weight: 900; letter-spacing: -1px } — letter-spacing: -1px có tác dụng gì? Khi nào dùng giá trị âm? Typography▸

Letter-spacing âm thu hẹp khoảng cách giữa các ký tự. Font size lớn (50px+) thường trông rời rạc — thu hẹp letter-spacing giúp chữ liền lạc, professional hơn. Đây là kỹ thuật typography phổ biến cho heading lớn.

49 .detail-category { text-transform: uppercase; letter-spacing: .08em } — tại sao dùng em cho letter-spacing thay vì px? Typography▸

em là đơn vị tương đối so với font-size của chính phần tử. Nếu font-size thay đổi, letter-spacing tự scale theo — chữ lớn hơn có khoảng cách lớn hơn. px cố định: chữ nhỏ có letter-spacing lớn trông kỳ. Best practice cho typography.

50 .item-name { line-height: 1.3 } và .detail-desc-short { line-height: 1.7 } — tại sao heading ngắn hơn line-height so với đoạn văn? Typography▸

Heading to thường 1 dòng hoặc rất ít dòng — line-height thấp (1.2-1.4) giúp chúng compact, tight. Đoạn văn dài cần line-height cao (1.6-1.8) để mắt dễ theo dõi từ cuối dòng sang đầu dòng tiếp. Đây là nguyên tắc typography cơ bản về readability.

Phần 8 — Màu sắc & Thiết kế hệ thống
51 Màu chủ đạo dùng trong toàn project là #4e5c34 (olive dark) và #6b7c4a (olive). Tại sao không dùng CSS custom property (--olive: #6b7c4a) cho màu này? Màu sắc▸

Nếu dùng CSS variable, đổi màu thương hiệu chỉ cần sửa 1 dòng. Hiện tại #6b7c4a xuất hiện hàng chục lần — nếu brand đổi màu phải sửa thủ công khắp nơi, dễ bỏ sót. Đây là điểm yếu về maintainability của codebase.

52 box-shadow: 0 4px 20px rgba(107,124,74,.08) — giải thích 4 giá trị của box-shadow. rgba(107,124,74,.08) là màu gì? Màu sắc▸

Thứ tự: x-offset, y-offset, blur-radius, spread-radius (tùy chọn), color. Ở đây: 0 ngang, 4px xuống, 20px mờ, không spread. rgba(107,124,74,.08) là màu olive (#6b7c4a) với opacity 8% — bóng đổ màu olive mờ thay vì bóng đen, hòa hợp với tone màu xanh lá của design.

53 background: linear-gradient(135deg, #4e5c34 0%, #6b7c4a 55%, #8a9a62 100%) trong khuyen_mai.css — giải thích 135deg và các color stop. Màu sắc▸

135deg = hướng diagonal từ trên trái xuống dưới phải (45° là hoàn toàn diagonal, 135° nghiêng ngược). Color stops: màu tối nhất ở 0%, mid-tone ở 55%, màu sáng nhất ở 100% — tạo hiệu ứng sáng dần từ trái sang phải. 3 stops cho gradient mượt mà hơn 2 stops.

54 .voucher-left { background: linear-gradient(180deg, #4e5c34, #6b7c4a) } — 180deg nghĩa là gradient theo hướng nào? Màu sắc▸

180deg = từ trên xuống dưới (top to bottom). 0deg = bottom to top, 90deg = trái sang phải, 270deg = phải sang trái. Cũng có thể viết to bottom thay cho 180deg.

55 lienhe.css dùng màu #f26522 (cam) cho button, trong khi toàn bộ web dùng màu olive #4e5c34. Đây có phải vấn đề design consistency không? Màu sắc▸

Đây là inconsistency — trang liên hệ có style riêng rẽ, không tuân thủ design system của phần còn lại. User chuyển từ trang sản phẩm sang trang liên hệ sẽ thấy màu button khác hẳn. Nên dùng chung class .btn-primary từ style.css.

Đây là bằng chứng nhiều người làm (3 thành viên nhóm) mà không thống nhất design system.
Phần 9 — Box Model & Sizing
56 * { box-sizing: border-box } trong admin.css — border-box khác content-box (mặc định) như thế nào? Ví dụ: phần tử width: 200px; padding: 20px. Box Model▸

content-box: width=200px là chiều rộng content, tổng chiều rộng = 200 + 20+20 (padding) = 240px. border-box: width=200px là TỔNG chiều rộng kể cả padding, content chỉ còn 160px. border-box dễ tính toán layout hơn nhiều.

57 .detail-image-box { flex: 0 0 420px } — nếu màn hình chỉ rộng 375px (iPhone SE), phần tử này rộng bao nhiêu? Giải pháp? Box Model▸

Vẫn cố định 420px — rộng hơn cả màn hình 375px → tràn ngang (horizontal overflow). Giải pháp: thêm max-width: 100% hoặc dùng media query để đổi thành flex: 1 (chiếm toàn bộ chiều rộng) trên mobile.

58 .cart-qty input { width: 44px } với type="number" — input number có spinner mặc định của browser. Code có ẩn spinner không? Cách ẩn bằng CSS? Box Model▸

Code không ẩn spinner — vẫn hiển thị mũi tên tăng/giảm của browser bên cạnh số. Cách ẩn: input[type=number]::-webkit-inner-spin-button, ::-webkit-outer-spin-button { -webkit-appearance: none } và input[type=number] { -moz-appearance: textfield } cho Firefox.

59 .btn-addcart:hover { background: #4e5c34;; transform: translateY(-2px) } — có lỗi syntax nào không? Browser xử lý thế nào? Box Model▸

Dấu chấm phẩy đôi ;; — lỗi syntax nhỏ. Browser bỏ qua (ignore) nó, không crash. CSS parser rất tolerant — lỗi nhỏ không làm hỏng toàn bộ rule. Tuy nhiên đây vẫn là code bẩn, nên fix. File code có nhiều chỗ bị lỗi ;; như vậy.

60 .qty-control { width: fit-content } — fit-content hoạt động như thế nào? Khác max-content và min-content? Box Model▸

fit-content: chiếm rộng vừa đủ nội dung, không vượt quá container. max-content: rộng tối đa để nội dung không xuống dòng (có thể vượt container). min-content: hẹp nhất có thể (wrap nhiều nhất). fit-content = min(max-content, available-space).

Phần 10 — Overflow & Visibility
61 .detail-image-box { overflow: hidden } kết hợp border-radius: 16px — tại sao cần overflow hidden? Overflow▸

Ảnh bên trong không có border-radius. Nếu container có border-radius nhưng không có overflow: hidden, ảnh sẽ "chọc thủng" các góc tròn — góc ảnh sẽ nhô ra khỏi container bo tròn. overflow: hidden cắt bỏ phần ảnh vượt ra ngoài border-radius.

62 nav style="overflow: visible !important" — tại sao phải dùng !important ở đây? Cách nào tốt hơn? Bẫy▸

Có rule nào đó (CSS hoặc framework) đang set overflow trên nav, khiến dropdown bị cắt. !important là "hack" để override. Cách tốt hơn: tìm ra rule nào đang gây overflow đó rồi xóa/sửa đúng source — tránh !important vì nó làm cascade khó debug.

!important = dấu hiệu của code đang "chữa ngọn" thay vì "chữa gốc".
63 .admin-sidebar { overflow-y: auto } — tại sao sidebar cần overflow-y: auto mà không phải scroll? Overflow▸

scroll: luôn hiển thị scrollbar dù nội dung ngắn — trông xấu khi menu chỉ có vài item. auto: scrollbar chỉ xuất hiện KHI CẦN — nội dung dài hơn container. UX tốt hơn và nhìn sạch hơn khi menu không đủ dài để scroll.

Phần 11 — Các câu hỏi bẫy & hay sai nhất
64 .footer { margin-left: 1000px } — giảng viên thấy dòng này trong code. Bạn giải thích như thế nào? Bẫy▸

Hardcode pixel tuyệt đối — bị lỗi trên màn hình không đúng 1440px. Logo có thể tràn ra ngoài màn hình nhỏ hơn hoặc không đến cạnh phải trên màn hình lớn hơn. Cách đúng: margin-left: auto trong flex container để đẩy logo về phải dynamically.

65 .left-menu ul li:hover { font-size: 16px } — vấn đề UX/layout gì xảy ra khi hover? Bẫy▸

Font-size thay đổi → element cao hơn → các item dưới bị đẩy xuống → layout shift. Các item xung quanh "nhảy" — người dùng mất cảm giác ổn định. Giải pháp không gây layout shift: font-weight: bold (ít thay đổi kích thước hơn) hoặc transform: scale(1.05).

66 Cả login lẫn register form đều action="login" method="post". Phân biệt bằng name="action" value="login/register". Bạn giải thích cơ chế này với giảng viên. Bẫy▸

Cả 2 form gửi đến cùng 1 Servlet (LoginServlet). Servlet đọc request.getParameter("action") để biết là "login" hay "register". Không cần 2 Servlet riêng. Button submit có name="action" — giá trị của button được submit được gửi kèm form.

67 .tab-btn { border-bottom: 3px solid transparent; margin-bottom: -2px } — trick gì đang xảy ra với margin âm này? Bẫy▸

Tab nav có border-bottom: 2px solid #d6ddc8. Tab button có border-bottom: 3px solid và margin-bottom: -2px. Khi active, border-bottom màu olive đè lên border nav — tạo hiệu ứng tab "liền" với content bên dưới. Technique kinh điển cho tab component.

68 .modal { display: none; position: fixed; ... display: flex } — modal dùng display: flex chứ không phải display: block. Lý do? Bẫy▸

Flex giúp căn giữa .modal-box cả ngang lẫn dọc bằng justify-content: center; align-items: center. Nếu dùng block, cần trick khác như margin: auto hoặc top: 50%; transform: translate(-50%, -50%) phức tạp hơn.

69 .btn-submit { display: block; margin: 0 auto } — tại sao cần display: block để margin: 0 auto căn giữa hoạt động? Bẫy▸

margin: 0 auto chỉ hoạt động trên block-level element có chiều rộng xác định. Button mặc định là inline-block. display: block khiến nó chiếm full width, sau đó margin: auto chia đều khoảng trống 2 bên để căn giữa.

Lỗi kinh điển: quên display block rồi hỏi tại sao margin auto không căn giữa.
70 .detail-image-box img { display: block } — tại sao ảnh cần display: block? Nếu không có thì bị lỗi gì? Bẫy▸

Ảnh mặc định là inline — inline elements có "baseline" nên có khoảng trắng nhỏ (3-4px) phía dưới. Container sẽ cao hơn ảnh vài px tạo khoảng trắng bí ẩn bên dưới ảnh. display: block loại bỏ baseline gap này.

Bẫy hay hỏi trong vấn đáp — "tại sao container cao hơn ảnh?".
71 .promo-hero { position: relative; overflow: hidden } — overflow: hidden ở đây phục vụ mục đích gì? Không phải để ẩn nội dung? Bẫy▸

Để cắt các hình tròn trang trí (::before, ::after) không tràn ra ngoài hero banner. Các hình tròn to (360px) kết hợp với position: absolute; top: -90px; right: -90px sẽ vượt ra ngoài container nếu không có overflow hidden.

72 .cd-box { backdrop-filter: blur(6px) } — backdrop-filter có browser support tốt không? Cần fallback gì? Bẫy▸

Hỗ trợ tốt trên Chrome, Safari nhưng Firefox cần flag. Cần fallback: đặt background màu solid có opacity trước, rồi override bằng backdrop-filter. Ví dụ: background: rgba(255,255,255,0.2); @supports (backdrop-filter: blur(1px)) { background: transparent; backdrop-filter: blur(6px); }.

73 .voucher-code { font-family: monospace } — tại sao mã voucher (GIAM10, DECOR20) dùng monospace font? Bẫy▸

Monospace: mỗi ký tự rộng bằng nhau → dễ đọc và so sánh code. "GIAM10" và "GIAM1O" (số 0 vs chữ O) dễ phân biệt hơn. Convention dùng monospace cho code, password, serial number, mã coupon từ thời terminal.

74 Code có border: 2px dashed #d6ddc8 trên voucher card và border: 1px solid #e2e8d8 trên product card. Tại sao 2 loại border khác nhau? Bẫy▸

Design có chủ ý — voucher dùng dashed để gợi nhớ hình vé/coupon vật lý có đường cắt lỗ chấm. Product card dùng solid là border thông thường. Sử dụng visual metaphor trong thiết kế để người dùng liên tưởng đến đối tượng thực tế.

75 .form-box textarea { resize: none } trong lienhe.css — nếu không có resize: none thì user làm được gì? Tại sao đôi khi nên cho phép resize? Bẫy▸

Không có → user có thể kéo góc dưới phải để thay đổi kích thước textarea, phá vỡ layout. resize: none ngăn điều này. Tuy nhiên resize: vertical (chỉ cho phép kéo dọc) là UX tốt hơn — user có thể xem được toàn bộ nội dung đã gõ mà không làm hỏng layout ngang.

Phần 12 — Câu hỏi về Object-fit, Background & Image
76 .item-img img { object-fit: cover } với container 72x72px nhưng ảnh gốc có thể là 500x700px. Phần nào của ảnh được giữ lại? Image▸

Cover scale ảnh đủ để phủ kín container, cắt phần thừa. Mặc định object-position: 50% 50% (center) — giữ phần giữa ảnh. Ảnh 500x700 → scale down để chiều ngắn (500px) phủ 72px → tỉ lệ 0.144 → chiều dài 700*0.144=101px → cắt 2 đầu dư 29px mỗi phía.

77 .featured-deal .deal-img { width: 44%; height: 320px; object-fit: cover } — 44% là tương đối so với gì? Image▸

Tương đối so với chiều rộng của flex container cha (.featured-deal). Phần tử flex với % tính theo container. Ảnh chiếm 44% width card, phần còn lại (khoảng 56%) là deal-body text. Tỷ lệ 44/56 gần với tỷ lệ vàng.

78 .hero-banner { background-image: url(...); background-size: cover; background-position: center } — phân biệt background-size cover vs contain. Image▸

cover: scale ảnh để phủ kín container, cắt phần thừa — không có khoảng trắng nhưng ảnh có thể bị crop. contain: scale ảnh để toàn bộ ảnh vừa vào container — không cắt nhưng có thể có khoảng trắng 2 bên (letterbox). Banner dùng cover để tránh khoảng trắng xấu.

79 .promo-card-img { width: 100%; height: 210px; object-fit: cover; display: block } — tại sao cần display: block trên thẻ <img>? Image▸

Giống câu 70 — ảnh là inline element có baseline gap phía dưới. display: block loại bỏ khoảng trắng bí ẩn dưới ảnh. Nếu không có, card sẽ có thêm 3-4px trống phía dưới ảnh làm mất sạch thiết kế.

Phần 13 — Specificity & Cascade
80 Style inline style="color: white" trên nhiều thẻ HTML — specificity của inline style là bao nhiêu? Chỉ có gì override được nó? Specificity▸

Inline style có specificity (1,0,0,0) — cao hơn mọi selector CSS ngoại trừ !important. Chỉ !important trong stylesheet override được. Đây là lý do không nên dùng inline style nhiều — rất khó override từ CSS file.

81 Breadcrumb được định nghĩa trong cả giohang.css và chitietsp.css với style gần giống nhau. Cái nào sẽ thắng nếu cả 2 file được load cùng lúc? Specificity▸

Specificity của 2 rule bằng nhau → CSS cascade áp dụng: file được link SAU thắng (last rule wins). Thứ tự <link> trong HTML quyết định. Đây là ví dụ về CSS duplication — nên đặt breadcrumb style chung vào style.css thay vì lặp.

82 table { border-collapse: collapse } trong admin.css — border-collapse có 2 giá trị, giải thích sự khác nhau và tại sao cần collapse? Specificity▸

separate (mặc định): mỗi cell có border riêng → giữa 2 cell có 2 border sát nhau trông dày và xấu. collapse: cells chia sẻ border → border giữa 2 cell chỉ có 1 đường. Table data trông clean hơn nhiều với collapse.

Phần 14 — Câu hỏi tổng hợp / nâng cao
83 .hero-eyebrow { background: rgba(255,255,255,0.15); border: 1px solid rgba(255,255,255,0.3) } — kỹ thuật glassmorphism nhẹ này có những property gì? Tại sao dùng rgba thay vì màu solid? Nâng cao▸

Glassmorphism cơ bản: background semi-transparent, border semi-transparent, thường thêm backdrop-filter blur. Dùng rgba thay solid vì nền gradient phía sau cần "xuyên qua" — màu solid sẽ che hẳn gradient. rgba cho phép gradient hiển thị qua background, tạo cảm giác chiều sâu.

84 .admin-content form input:focus { box-shadow: 0 0 0 3px rgba(78,92,52,0.1) } — kỹ thuật focus ring này hoạt động thế nào? Tại sao không dùng border-width? Nâng cao▸

box-shadow: 0 0 0 3px với blur=0 và spread=3px = một ring bằng phẳng bao quanh element, không làm thay đổi kích thước element (không trigger layout reflow). Đổi border-width sẽ thay đổi kích thước element → layout shift. Đây là modern focus ring pattern.

85 .hero-cta { border-radius: 50px } — 50px trên một button không phải hình tròn tạo ra hình gì? Khác border-radius: 50% thế nào? Nâng cao▸

50px trên button cao ~50px tạo hình "pill" (viên thuốc) — góc bo tròn hoàn toàn ở 2 đầu. Miễn là border-radius >= height/2 thì sẽ tạo pill shape bất kể button rộng hay hẹp. 50% trên rectangle tạo ellipse (bầu dục), khác pill.

86 .new-sidebar { align-self: flex-start } — align-self khác align-items ở điểm gì? Flexbox▸

align-items: đặt trên flex container, áp dụng cho TẤT CẢ flex items. align-self: đặt trên từng flex item riêng lẻ, override align-items cho item đó. Sidebar dùng align-self: flex-start để chỉ riêng sidebar được cao vừa nội dung, dù container có align-items khác.

87 .stat-num { display: block } trên <span> — tại sao cần display: block trên span? Nâng cao▸

Span mặc định là inline — inline elements không xuống dòng. display: block cho phép .stat-num và .stat-label xếp dọc (mỗi cái 1 dòng riêng) thay vì nằm ngang cùng nhau. Dùng display: block trên inline element là pattern rất phổ biến.

88 .pay-badge { border: 1px solid #e2e8d8; color: #6b7280 } — không có background color, tại sao badge vẫn có màu nền? Nâng cao▸

Background của .payment-row { background: #f8f9fa } là cha. Badge không có background riêng → trong suốt → kế thừa màu nền của cha (#f8f9fa). CSS background không được kế thừa theo cascade thông thường nhưng vì badge transparent nên nhìn xuyên qua thấy màu nền cha.

89 .cart-header span:nth-child(2), span:nth-child(3), span:nth-child(4) { text-align: center } — selector này có bug không? Bẫy▸

Có bug tiềm ẩn: span:nth-child(3) sau dấu phẩy là selector ĐỘC LẬP — nghĩa là TẤT CẢ span là con thứ 3 trong trang, không phải chỉ trong .cart-header. Đúng phải viết: .cart-header span:nth-child(2), .cart-header span:nth-child(3), .cart-header span:nth-child(4).

Lỗi selector sau dấu phẩy rất hay gặp, giảng viên thích hỏi về cái này.
90 .admin-wrapper { display: flex; min-height: 100vh } — tại sao dùng min-height thay vì height: 100vh? Nâng cao▸

height: 100vh cố định chiều cao đúng viewport — nếu content dài hơn viewport sẽ bị overflow (cut off). min-height: 100vh đảm bảo ít nhất cao bằng viewport, nhưng có thể cao hơn nếu content nhiều — trang tự scroll đúng cách.

Phần 15 — 10 câu cuối: Kiến thức mở rộng
91 Tại sao mỗi trang JSP link cả style.css và file CSS riêng (giohang.css, chitietsp.css...)? Không gộp vào 1 file được không? Kiến trúc▸

Tách file theo trang: trang giỏ hàng chỉ load giohang.css, trang chi tiết load chitietsp.css — giảm lượng CSS không cần thiết. Tuy nhiên 1 file lớn có thể cache tốt hơn. Trong production thường dùng CSS bundler (webpack) gộp và minify. Cách làm trong project là chấp nhận được cho project học.

92 font-weight: 900 và font-weight: 800 đều được dùng. Montserrat có hỗ trợ những weight này không? Nếu không thì browser làm gì? Typography▸

Montserrat được load với wght@400;500;600;700;800;900 trong Google Fonts URL — có đủ cả 900. Nếu weight không có, browser dùng font-weight gần nhất có trong font. Ví dụ: nếu chỉ có 400 và 700, weight 900 sẽ dùng 700.

93 Code load Font Awesome từ CDN: cdnjs.cloudflare.com. Nếu user offline thì icon hiển thị thế nào? Cách khắc phục? Kiến trúc▸

Icon sẽ không hiển thị — chỉ thấy ký tự lạ hoặc khoảng trống. Khắc phục: download Font Awesome về serve locally (trong thư mục web/css hoặc web/fonts). Trong môi trường production hoặc khi demo không có internet, đây là vấn đề nghiêm trọng.

94 .promo-hero { padding: 56px 50px } — padding lớn như vậy trên mobile (320px width) chiếm bao nhiêu % chiều rộng? Có vấn đề gì không? Responsive▸

Padding ngang 50px * 2 = 100px chiếm 100/320 = 31% chỉ để làm padding. Content chỉ còn 220px — rất chật. Nên dùng responsive padding: padding: 30px 20px trên mobile qua media query. Hoặc dùng padding: 56px clamp(20px, 5vw, 50px).

95 Nhiều chỗ trong code dùng màu text #6b7280 — đây là màu gì trong hệ màu nào? Cách nào giúp maintain màu tốt hơn? Màu sắc▸

#6b7280 là gray-500 trong Tailwind CSS color system — màu xám trung bình cho text phụ. Dùng nhiều chỗ nhưng không có CSS variable → khó maintain. Best practice: khai báo :root { --text-muted: #6b7280; --text-dark: #2c2c2c; } rồi dùng var(--text-muted) khắp nơi.

96 animation: slideDown 0.25s ease — giải thích ease. Còn những timing function nào khác? Cái nào thường dùng nhất? Animation▸

ease: bắt đầu chậm, tăng tốc, rồi chậm lại (tự nhiên nhất). Các giá trị khác: linear (đều), ease-in (bắt đầu chậm), ease-out (kết thúc chậm), ease-in-out (cả 2 đầu chậm), cubic-bezier(x1,y1,x2,y2) (tùy chỉnh). ease và ease-out dùng nhiều nhất vì tự nhiên nhất.

97 Trong admin.css có cả .btn generic và .admin-content form .btn specific. Khi cả 2 áp dụng lên cùng 1 element thì cái nào thắng? Bẫy▸

.admin-content form .btn có specificity (0,3,0) — 3 class selectors. .btn đơn giản chỉ có (0,1,0). Rule phức tạp hơn thắng, bất kể thứ tự trong file. Đây là lý do cần hiểu specificity, không chỉ đặt style ở cuối file để "override".

98 text-decoration: line-through trên .deal-price-old — ngoài line-through, còn những giá trị nào khác của text-decoration? Typography▸

none, underline, overline, line-through. Có thể kết hợp: underline overline. CSS3 mở rộng thêm: text-decoration-color, text-decoration-style (solid/dashed/dotted/wavy), text-decoration-thickness. Ví dụ: text-decoration: underline wavy red.

99 Nhiều element dùng cursor: pointer. Khi nào NÊN và KHÔNG NÊN dùng cursor pointer? UX/CSS▸

NÊN: tất cả element clickable không phải link/button mặc định — div, span, icon có onclick. KHÔNG NÊN: input text (nên là cursor text/default), disabled elements (cursor: not-allowed), non-clickable decorative elements. Code đặt cursor: pointer trên button là redundant (button đã có pointer mặc định) nhưng không sai.

100 Nhìn tổng thể, điểm mạnh và điểm yếu lớn nhất về CSS trong toàn bộ project của nhóm bạn là gì? Tổng hợp▸

Điểm mạnh: Design system màu olive nhất quán, sử dụng flexbox/grid đúng cách, animation mượt với transform/opacity, sticky positioning hợp lý, có breadcrumb và breadcrumb hover states. Điểm yếu: Thiếu responsive trên phần lớn component, không dùng CSS custom property, duplicate code giữa các file, vài hardcode pixel bất hợp lý, inconsistency giữa lienhe.css và phần còn lại.

Trả lời câu này thành thật và phân tích rõ ràng sẽ gây ấn tượng tốt hơn là chỉ khen code của mình.
